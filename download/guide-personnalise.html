Section 02 · Bot Telegram
Créer le bot via @BotFather
Le bot Telegram sera l'administrateur des deux canaux et publiera les messages automatiquement. Tu le crées une seule fois via le compte officiel @BotFather (vérifié avec ✅). Ce bot recevra ensuite le droit de poster sur les canaux FR et EN.
Étapes de création
  1. Ouvre Telegram et cherche @BotFather (compte vérifié avec ✅)
  2. Envoie la commande /newbot
  3. BotFather te demande un nom. Choisis : P2P Scout Bot
  4. BotFather te demande un username (doit finir par "bot"). Choisis : P2PScoutBot
  5. BotFather te renvoie un token API au format 123456789:ABCdefGHIjklMNOpqrSTUvwxYZ. Copie-le immédiatement.
  6. Stocke ce token dans une note sécurisée (1Password, Bitwarden, ou vault chiffré). Tu en auras besoin pour la section 7.

# Test rapide que le bot répond curl "https://api.telegram.org/bot<TON_TOKEN>/getMe" # Réponse attendue : # {"ok":true,"result":{"id":123456789,"is_bot":true, # "first_name":"P2P Scout Bot","username":"P2PScoutBot"}}

SÉCURITÉ CRITIQUE : Ne partage JAMAIS ce token. Si tu le fuites accidentellement (commit Git, screenshot, etc.), reset-le immédiatement via la commande /revoke dans @BotFather. Quiconque possède le token contrôle ton bot.

Section 06 · IDs des canaux
Récupérer les IDs des 2 canaux
Pour que le bot puisse publier sur un canal, tu dois connaître son chat_id (identifiant numérique négatif). Telegram ne l'affiche pas dans l'interface — il faut le récupérer via l'API. La méthode la plus simple : utiliser le bot @userinfobot ou interroger l'API Telegram directement avec curl.
Méthode 1 — via @userinfobot (le plus simple)
  1. Ajoute @userinfobot à ton canal FR comme admin (avec droit de lecture des messages)
  2. Poste n'importe quel message dans le canal
  3. @userinfobot répond automatiquement avec l'ID du canal (format : -1001234567890)
  4. Note cet ID pour le canal FR, retire le bot du canal
  5. Répète pour le canal EN → tu obtiens un 2ème ID
  6. Retire @userinfobot des deux canaux
Méthode 2 — via curl (plus rapide, sans bot tiers)
bash — récupérer le chat_id
# 1. Récupère le chat_id du canal FR # Remplace @ArbitrageP2P_FR par le username PUBLIC de ton canal FR curl "https://api.telegram.org/bot<TON_TOKEN>/getChat?chat_id=@ArbitrageP2P_FR" # Réponse : # {"ok":true,"result":{"id":-1001234567890, # "title":"Arbitrage P2P — Signaux Gratuits 🇫🇷",...}} # 2. Récupère le chat_id du canal EN curl "https://api.telegram.org/bot<TON_TOKEN>/getChat?chat_id=@P2PArbitrage_EN" # Réponse : # {"ok":true,"result":{"id":-1001234567891, # "title":"P2P Arbitrage — Free Signals 🌍",...}} # 3. Note les 2 IDs (TOUJOURS négatifs, commencent par -100) # TELEGRAM_CHANNEL_FR_ID = -1001234567890 # TELEGRAM_CHANNEL_EN_ID = -1001234567891
⚠️
Format des IDs : Les IDs de canaux sont TOUJOURS négatifs et commencent par -100. Si tu vois un ID positif, c'est celui d'un utilisateur, pas d'un canal. Vérifie que le bot est bien admin du canal avant de tester, sinon tu obtiendras une erreur 403 Forbidden.

Section 08 · Crons externes
4 méthodes pour bypass la limite cron Vercel Hobby
Vercel Hobby = 1 cron job max, fréquence quotidienne (1x/jour). Insuffisant pour scanner toutes les 5 minutes. Le hack : utiliser plusieurs services de cron externes gratuits qui frappent l'endpoint /api/scan/trigger. Cet endpoint est idempotent grâce à un verrou distribué dans Neon (table ScanLock) + un debounce de 2 minutes. Résultat : scan effectif toutes les 1-5 min, gratuit, sans dépendre du cron Vercel.
🔒
Idempotence garantie : Combine les 4 méthodes ci-dessous pour une redondance maximale. Même si 3 triggers arrivent simultanément, le verrou ScanLock permet à 1 seul de s'exécuter — les 2 autres retournent skipped: "locked". Le debounce de 2 min évite aussi le spam.
Méthode A — UptimeRobot
5 min · 50 monitors
Gratuit : Oui · Headers custom : Non (plan gratuit) · Latence min : 5 min
Va sur uptimerobot.com → Sign up → Add New Monitor → Type "HTTP(s)" → URL : https://ton-app.vercel.app/api/scan/trigger?run=1 → Interval : 5 minutes. Comme UptimeRobot ne supporte pas les headers custom en gratuit, soit tu ne définis pas CRON_API_KEY (accès ouvert), soit tu crées une variante d'endpoint qui accepte la clé en query string. Idéal pour démarrer sans friction.
Méthode B — cron-job.org
1-5 min · headers OK
Gratuit : Oui · Headers custom : Oui · Recommandé
Va sur cron-job.org → Sign up → Create Cronjob → URL : https://ton-app.vercel.app/api/scan/trigger → Schedule : Every 5 minutes (*/5 * * * *) → Method : POST → Headers : X-API-Key: <ta-cle>. C'est la méthode recommandée car elle supporte les headers custom gratuitement.
Méthode C — GitHub Actions
5 min · 2000 min/mois
Gratuit : Oui (2000 min/mois) · Headers custom : Oui · Redondance
Crée .github/workflows/trigger_scan.yml dans ton repo. Le workflow fait un curl POST vers ton endpoint toutes les 5 minutes. Limite : 2000 min/mois = ~33 runs/jour si chaque run dure 2 min. À combiner avec cron-job.org pour atteindre le rythme 5 min complet.
Méthode D — Upstash QStash
500 msg/jour · retry garanti
Gratuit : Oui (500 msg/jour) · Le plus élégant · Retry auto
QStash = queue serverless avec scheduling intégré. Va sur console.upstash.com → QStash → crée un schedule avec cron: "*/5 * * * *" qui POST vers ton endpoint. QStash garantit la livraison (retry auto si erreur 500), supporte les headers custom, et c'est gratuit jusqu'à 500 messages/jour = 1 toutes les 3 min en moyenne.
Configuration cron-job.org (Méthode B — recommandée)
cron-job.org — formulaire de configuration
# Champs à remplir sur cron-job.org : Title : P2P Scout Scan 5min URL : https://ton-app.vercel.app/api/scan/trigger Request Method : POST Execution Schedule : Every 5 minutes (cron: */5 * * * *) Notifications : Email on failure # Headers (section "Headers") : X-API-Key : <ta-cle-CRON_API_KEY> Content-Type : application/json # Body (vide — pas besoin de payload) # Timeout : 60 seconds # Save → le scan tourne toutes les 5 min, gratuit
Configuration vercel.json — cron quotidien pour le rapport
json — vercel.json
{ "crons": [ { "path": "/api/daily-report", "schedule": "0 8 * * *" } ] } # Le cron Vercel (1/jour, 8h UTC) sert UNIQUEMENT au rapport quotidien. # Le scanner est déclenché par les crons externes (Méthodes A-D).
